Практическое занятие 1: Жизненный цикл программного обеспечения

МДК.02.01 Технология разработки программного обеспечения

Цель занятия:

Освоить основные этапы жизненного цикла программного обеспечения (ЖЦ ПО) и научиться применять различные модели жизненного цикла на практике.

Теоретическая часть

1. Понятие жизненного цикла ПО

Жизненный цикл программного обеспечения — это период времени, который начинается с момента принятия решения о необходимости создания ПО и заканчивается в момент его полного изъятия из эксплуатации.

2. Основные этапы ЖЦ ПО

2.1. Анализ требований

  • Сбор и анализ требований заказчика
  • Определение целей и задач проекта
  • Формирование технического задания

2.2. Проектирование

  • Разработка архитектуры системы
  • Проектирование базы данных
  • Создание прототипов интерфейсов

2.3. Разработка

  • Написание кода
  • Модульное тестирование
  • Интеграция компонентов

2.4. Тестирование

  • Функциональное тестирование
  • Нагрузочное тестирование
  • Тестирование безопасности

2.5. Внедрение

  • Установка на рабочее место заказчика
  • Миграция данных
  • Обучение пользователей

2.6. Сопровождение

  • Исправление ошибок
  • Добавление нового функционала
  • Техническая поддержка

Практическая часть

Задание 1: Сравнение моделей ЖЦ ПО

Инструкция: Изучите характеристики различных моделей ЖЦ ПО и заполните сравнительную таблицу.

Модель Преимущества Недостатки Область применения
Водопадная (Waterfall) Простота и четкая структура; легко планировать и контролировать этапы; подходит для проектов с фиксированными требованиями. Жесткость — трудно вносить изменения после завершения этапов; риски проявляются поздно; неэффективна для проектов с изменяющимися требованиями. Стабильные проекты с хорошо определенными требованиями, такие как разработка крупных систем в государственном секторе или для критической инфраструктуры.
Итерационная Позволяет постепенно улучшать продукт через повторяющиеся циклы; раннее обнаружение ошибок; гибкость в адаптации к изменениям. Может быть сложнее управлять из-за повторений; требует больше ресурсов на планирование и интеграцию. Проекты с неопределенными или эволюционирующими требованиями, например, прототипирование или разработка сложных систем.
Спиральная Фокус на управлении рисками; комбинирует итеративный подход с анализом; гибкость для больших проектов. Высокая сложность и стоимость; требует значительной экспертизы и ресурсов. Крупные, рискованные проекты, такие как разработка ПО для оборонной промышленности или критических систем.
Agile Быстрая адаптация к изменениям; частые релизы и обратная связь от клиентов; повышение вовлеченности команды. Требует высокой дисциплины и культуры сотрудничества; может привести к хаосу без надлежащего управления; не подходит для строго регламентированных проектов. Гибкие проекты с быстрыми изменениями, стартапы, веб-разработка и приложения с короткими циклами выпуска.
V-образная Раннее и систематическое тестирование на каждом этапе; высокое качество через верификацию и валидацию; четкая связь между разработкой и тестированием. Жесткая структура, как у водопадной модели; трудно адаптироваться к изменениям; высокие затраты на тестирование. Проекты с высокими требованиями к качеству и безопасности, такие как аэрокосмическая отрасль, медицинские системы или программное обеспечение для транспорта.

Пример заполнения для Waterfall:

  • Преимущества: Простота управления, четкое планирование
  • Недостатки: Низкая гибкость, позднее тестирование
  • Область применения: Проекты с четкими требованиями

Задание 2: Разработка плана проекта

Сценарий: Компания разрабатывает мобильное приложение для доставки еды. Необходимо создать план проекта по этапам ЖЦ ПО.

Требуется:

  1. Определить основные этапы проекта
  2. Распределить задачи по этапам
  3. Оценить сроки выполнения
  4. Определить ключевые результаты каждого этапа

Шаблон плана проекта:

План разработки мобильного приложения "FoodDelivery"
  1. Анализ требований (1-2 недели)
    • Интервью с заказчиком
    • Анализ рынка и конкурентов
    • Формирование технического задания
    • Определение целевой аудитории

    Результат: Утвержденное техническое задание

  2. Проектирование (2 недели)
    • Разработка архитектуры приложения
    • Проектирование базы данных
    • Создание wireframe интерфейсов
    • Выбор технологического стека

    Результат: Технический дизайн-документ

  3. Разработка (6-8 недель)
    • Настройка среды разработки
    • Реализация backend API
    • Разработка мобильного приложения
    • Интеграция с платежными системами

    Результат: Рабочий прототип приложения

  4. Тестирование (2 недели)
    • Функциональное тестирование
    • Тестирование пользовательского интерфейса
    • Нагрузочное тестирование
    • Тестирование безопасности

    Результат: Отчет о тестировании

  5. Внедрение (1 неделя)
    • Публикация в App Store/Google Play
    • Настройка production-окружения
    • Миграция данных (если требуется)

    Результат: Приложение доступно для пользователей

  6. Сопровождение (постоянно)
    • Мониторинг работы приложения
    • Исправление обнаруженных ошибок
    • Добавление нового функционала
    • Техническая поддержка пользователей

    Результат: Стабильно работающее приложение

Задание 3: Выбор модели ЖЦ ПО

Сценарии проектов:

Проект A: Система управления банковскими операциями

  • Требования: четкие, неизменные
  • Сроки: жесткие
  • Бюджет: фиксированный
  • Риски: высокие (финансовые потери)

Проект B: Социальная сеть для фотографов

  • Требования: изменчивые
  • Сроки: гибкие
  • Бюджет: может меняться
  • Риски: средние

Проект C: Мобильная игра

  • Требования: частично определены
  • Сроки: сжатые
  • Бюджет: ограниченный
  • Риски: высокие (конкуренция)

Задание: Для каждого проекта обоснуйте выбор модели ЖЦ ПО и опишите, как будет организован процесс разработки.

Пример решения для Проекта A:

Выбранная модель: V-образная модель

Обоснование:

  • Высокие требования к качеству и надежности
  • Четкие и неизменные требования
  • Необходимость строгого контроля качества
  • Высокие риски финансовых потерь

Организация процесса:

  1. Детальный анализ и формализация требований
  2. Строгое соответствие этапов тестирования этапам разработки
  3. Обязательная верификация и валидация на каждом этапе
  4. Формальные процедуры приемки

Задание 4: Кейс-анализ

Ситуация: Команда разработчиков работает над проектом CRM-системы. Через 3 месяца после начала разработки заказчик решил полностью изменить концепцию продукта.

Вопросы для анализа:

  1. Какие проблемы возникнут при использовании водопадной модели?
  2. Как Agile-подход поможет справиться с изменениями?
  3. Какие риски необходимо учитывать при смене требований?
  4. Как организовать процесс управления изменениями?

Шаблон для ответа:

## Анализ ситуации

Проблемы при водопадной модели:

  • Необходимость переделывать уже завершенные этапы
  • Потеря времени и ресурсов
  • Конфликты с изначальным планом проекта

Преимущества Agile-подхода:

  • Возможность быстрой адаптации к изменениям
  • Поэтапная разработка с регулярной обратной связью
  • Гибкое планирование спринтов
